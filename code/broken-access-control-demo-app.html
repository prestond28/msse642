#1 OWASP 2025

Broken access control

The #1 most critical web application security risk, according to OWASP's 2021 and 2025 Top 10.

CWE-284 High severity Very common
Core concept

What is it?

Access control decides who can do what in an application. When it's broken, users can act outside their intended permissions — accessing other users' data, admin features, or protected resources they were never supposed to reach.

The bug is almost always the same: the server trusts the client, or forgets to check authorization at all.

Attack types

How attackers exploit it

1
IDOR — Insecure Direct Object Reference
Changing an ID in a URL to access someone else's record. e.g. /orders/42 → /orders/99
2
Privilege escalation
A regular user accessing admin-only pages or endpoints, either by guessing the URL or manipulating request data.
3
Forced browsing
Navigating directly to a URL like /admin/users without being authorized. The UI hides the link, but the server never checks.
4
HTTP method manipulation
Switching from GET to PUT or DELETE to bypass read-only restrictions on an endpoint.
5
Missing function-level checks
APIs that rely on the UI to hide sensitive actions — but don't verify authorization when the API is called directly.
Real-world impact

Why it matters

Exploiting broken access control lets attackers read private user data, modify or delete records, impersonate other users, perform admin actions, and take over entire accounts. It topped the OWASP list because it's simultaneously very common and very severe — and often trivially easy to exploit.

How to fix it

Four principles that eliminate most broken access control vulnerabilities.

Principle 1

Deny by default

If a route has no explicit access rule, refuse it. Never assume access is allowed — always require it to be granted.

# Bad — allow unless explicitly blocked
if user.role == "admin": show_page()

# Good — deny unless explicitly allowed
if not user.can("view_admin"): return 403
Principle 2

Always check ownership server-side

Never trust IDs from the client. Scope every database query to the authenticated user so it's impossible to access records belonging to others.

# Bad — trusts the ID from the URL
order = Order.find(params[:id])

# Good — scoped to the current user
order = current_user.orders.find(params[:id])
Principle 3

Never trust client-supplied roles

Roles and permissions must come from the server session, set at login — never from a form field, cookie value, or request body the user controls.

# Bad — attacker controls this
role = request.body["role"]

# Good — set by the server at login
role = session["role"]
Principle 4

Log and alert on failures

Every access control failure should be logged. Repeated failures from the same user or IP are a strong signal of an active attack — and should trigger rate limiting or alerts.

if not authorized:
  log.warn(f"Access denied: {user.id} → {route}")
  alert_if_threshold_exceeded(user.id)
  return 403
